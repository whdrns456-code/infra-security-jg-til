📝 1. 오늘의 학습 주제 및 핵심 내용
주요 주제: DNS Server 이중화(Master/Slave) 및 로컬 도메인 권한 설정

핵심 개념 1: BIND Zone Transfer (존 전송)

내용: Master DNS(10.33)의 레코드를 Slave DNS(10.34)가 자동으로 동기화하는 메커니즘. Serial 번호가 증가해야만 Slave가 변경 사항을 감지하고 가져옴.

핵심 개념 2: DNS Resolution 우선순위 (/etc/resolv.conf)

내용: 시스템이 도메인을 찾을 때 참조하는 네임서버의 순서. 로컬 서버에서 정의한 도메인을 우선 인식시키려면 공인 DNS(8.8.8.8)보다 내부 DNS IP를 상단에 배치해야 함.

💻 2. 실습 및 구현 내용
✅ 실습 목표
BIND를 이용한 Master/Slave DNS 서버 구축 및 로드밸런서(Nginx) VIP 연결

🛠️ 구현 결과
Bash
# [Master: /etc/named.rfc1912.zones]
zone "jonggun.com" IN {
    type master;
    file "jonggun.com.db";
    allow-transfer { 192.168.10.34; }; # Slave에게만 전송 허용
};

# [Slave: /etc/named.rfc1912.zones]
zone "jonggun.com" IN {
    type slave;
    file "slaves/jonggun.com.db";
    masters { 192.168.10.33; };        # Master 주소 지정
};

# [Zone File: /var/named/jonggun.com.db]
$TTL 1D
@       IN SOA  ns1.jonggun.com. root.jonggun.com. (
                2026030101 ; Serial (반드시 증가시켜야 반영됨)
                1D 1H 1W 3H )
        IN NS   ns1.jonggun.com.
ns1     IN A    192.168.10.33
www     IN A    192.168.10.10   ; 로드밸런서 VIP
✨ 새롭게 알게 된 점
LVM 마운트의 중요성: 서버 재부팅 후 LVM 장치가 /shares에 마운트되지 않으면 NFS 공유 자체가 깨지며 Stale file handle 에러가 발생한다는 것을 확인함.

Slave 자동 생성: Slave 서버는 Zone 파일을 직접 만들지 않아도 Master로부터 동기화하여 slaves/ 폴더 내부에 자동으로 생성함.

🛠️ 3. 트러블슈팅 (Troubleshooting)
🔍 문제 발생 (Issue)
상황: DNS 설정을 마쳤음에도 www.jonggun.com 접속 시 ERR_CONNECTION_REFUSED 발생.

에러 메시지: nslookup 결과가 Address: 52.20.84.62 (외부 공인 IP)로 나타남.

💡 원인 분석 (Cause)
/etc/resolv.conf 파일에 구글 공인 DNS인 8.8.8.8이 최상단에 등록되어 있어, 로컬 DNS 서버(10.33)에 묻기도 전에 외부 인터넷의 jonggun.com 주소를 먼저 찾아옴.

🚀 해결 방법 (Solution)
/etc/resolv.conf를 수정하여 로컬 DNS 서버와 루프백 주소를 최상단으로 올림.

Bash
# /etc/resolv.conf 수정
nameserver 127.0.0.1
nameserver 192.168.10.33
nameserver 192.168.10.34
nameserver 8.8.8.8 # 외부용은 가장 마지막에
수정 후 nslookup www.jonggun.com 실행 시 정상적으로 내부 VIP인 192.168.10.10을 반환하는 것 확인.
