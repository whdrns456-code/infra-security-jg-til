**은영 프로젝트 일지 2026-02-07**

**🛠 AWS 클라우드 기반 가상 서버 구축 및 보안 배포**

- **개요**
    - 목표: Spring Boot 애플리케이션을 AWS EC2(Ubuntu) 환경에 배포하고, 외부 사용자가 안전하게 접속할 수 있는 인프라 환경 구축.
    - 환경: AWS EC2, Ubuntu 24.04 LTS, OpenJDK 17, Nginx
  

**주요 작업 및 설정 이유**

- **단계 - 작업 내용 - 설정 이유** 
- 인스턴스 생성
    - AWS EC2 가상 서버 생성 및 SSH Key(pem) 설정,"독립된 서버 자원을 확보하고, 비대칭키 방식을 통해 보안 접속 권한을 설정함."
- 런타임 구축
    - OpenJDK 17 설치,Spring Boot 애플리케이션 실행을 위한 표준 자바 실행 환경(JRE) 구축.
- 보안 그룹 설정 
    - "인바운드 규칙 최적화 (80, 22, 8080 포트)",22(SSH): 관리 통로 / 80(HTTP): 사용자 접속용 / 8080: 앱 테스트용.
- 리버스 프록시 
    - Nginx 설치 및 설정,[보안 핵심] 애플리케이션 포트 직접 접근을 차단하고 서버 포트를 은닉함.
- 무중단 실행
    - nohup 및 백그라운드(&) 실행,터미널 종료 후에도 서버가 24시간 서비스되도록 설정함.
  

**보안 엔지니어로서의 관점**
- 🛡️ 포트 은닉 (Port Hiding): 
    - 초기에는 8080 포트로 직접 접속했으나, 보안 취약점을 줄이기 위해 Nginx 리버스 프록시를 도입하여 외부 노출을 차단했습니다.

- 🔐 최소 권한 원칙: 
    - 서비스 안정화 후 AWS Security Group에서 불필요한 포트(8080)를 닫고 
    - 꼭 필요한 서비스 포트(80)만 개방하여 **공격 표면(Attack Surface)**을 최소화했습니다.

- 📝 로그 관리: nohup.out을 통해 서버 내부 로그를 파일로 기록하여 서비스 상태 모니터링 및 장애 대응의 기초를 마련했습니다.
  

 **사용 기술 (Tech Stack)**
    - Cloud: AWS EC2
    - OS: Ubuntu 24.04 LTS
    - Web Server: Nginx (Reverse Proxy)
    - Language: Java 17 (OpenJDK)
    - Framework: Spring Boot  

  
  
**Trouble Shooting (문제 해결 경험)**
    - ❌ 문제 1: 프로세스 독점으로 인한 터미널 제어 불가
    - 현상: 서버 실행 후 터미널이 로그 출력에 점유되어 추가 명령 입력 불가.

- 해결: Ctrl + C로 중단 후, nohup과 & 연산자를 사용하여 백그라운드 프로세스로 전환.

    - ❌ 문제 2: 리버스 프록시 설정 시 에디터 조작 미숙
    - 현상: nano 에디터 수정 중 저장 프로세스(Ctrl + O 후 Enter) 누락으로 인한 먹통 현상.

- 해결: 세션 재접속 후 **sudo nginx -t**를 통한 문법 검증 및 systemctl restart로 안전한 설정 반영.
