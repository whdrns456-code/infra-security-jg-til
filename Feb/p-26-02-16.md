## 💻 2. 실습 및 구현 내용
### ✅ 실습 목표
- 사용자의 비밀번호를 DB에 평문으로 저장하지 않고, 보안 알고리즘을 적용하여 저장 및 검증하는 로그인 로직 구현
- 정보 노출 최소화 로그인 실패 원인을 구체적으로 제공함으로써 발생할 수 있는 잠재적인 보안 위협을 사전에 방지하고자 
`아이디 또는 비밀번호가 존재하지 않습니다.`메세지 통일 .
-

### 🛠️ 구현 결과
```java```
// UserService.java - 보안 로직 핵심
public User login(String userId, String userPw) {
    // 1. 사용자 조회 (계정 열거 공격 방지를 위해 에러 메시지 통합)
    User user = userRepository.findByUserId(userId)
            .orElseThrow(() -> new IllegalArgumentException("아이디 또는 비밀번호가 존재하지 않습니다."));

    // 2. BCrypt를 활용한 암호문 비교 (내장된 솔트값 자동 인식)
    if (!passwordEncoder.matches(userPw, user.getUserPw())) {
        throw new IllegalArgumentException("아이디 또는 비밀번호가 존재하지 않습니다.");
    }

    // 3. 계정 활성화 상태 확인
    if (!"ACTIVE".equals(user.getAccountStatus())) {
        throw new IllegalStateException("비활성화된 계정입니다.");
    }
    return user;
}

### ✨ 새롭게 알게 된 점
BCrypt의 내부 메커니즘: 단순히 일방향 해시 함수를 사용하는 줄 알았으나, 
무차별 대입 공격(Brute-force)과 레인보우 테이블 취약점을 보완하기 위해 
**솔팅(Salting)**과 **키 스트레칭(Key Stretching)**이 자동으로 적용된다는 점을 학습함.

Salting의 원리: 동일한 비밀번호를 입력해도 암호화 시점에 따라 서로 다른 해시값이 생성되므로,
공격자가 미리 계산된 해시 테이블(레인보우 테이블)을 사용할 수 없게 만든다는 점이 보안상 매우 중요하다는 것을 깨달음.
