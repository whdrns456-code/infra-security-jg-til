## 📝 1. 오늘의 학습 주제 및 핵심 내용
- **주요 주제: Nginx GeoIP2를 활용한 국가별 접속 제어 및 보안 강화**
- **핵심 개념 1: GeoIP2 (MaxMind)**
    - 내용: 접속자의 IP 주소를 전 세계 국가/도시 데이터베이스와 대조하여 지리적 위치 정보를 식별하는 기술.
- **핵심 개념 2: Nginx Map Module**
    - 내용: GeoIP2에서 추출한 국가 코드를 변수에 매핑하여, 특정 국가(CN, KP 등)에 대해 조건부 접속 허용/차단($allowed_country) 정책을 수립함.

## 💻 2. 실습 및 구현 내용
### ✅ 실습 목표
- MaxMind 데이터베이스를 연동하여 보안 위협이 높은 국가(중국, 북한 등)의 유입을 L7 계층에서 선제적으로 차단한다.

### 🛠️ 구현 결과
```nginx
# [/etc/nginx/nginx.conf]

# [1] GeoIP2 국가 식별 및 매핑 설정
# MaxMind의 GeoLite2 데이터베이스를 참조하여 접속자의 IP를 국가 코드로 변환하고,
# 변환된 국가 코드에 따라 서비스 허용 여부($allowed_country)를 결정합니다.

map $geoip2_data_country_code $allowed_country {
    default yes;  # 기본적으로 모든 국가의 접속을 허용합니다.

    # [차단 리스트 구성]
    # 서비스 대상이 아니며, 보안 위협(Scanning, Brute Force 등)이 빈번한 국가를 지정합니다.
    CN      no;   # 중국(China)으로부터의 모든 유입 차단
    KP      no;   # 북한(North Korea)으로부터의 모든 유입 차단
}

# [2] 실제 차단 로직 적용 (Server Block 내)
# if ($allowed_country = no) {
#     return 403; # 허용되지 않은 국가 접속 시 403 Forbidden 반환
# }
```

## ✨ 새롭게 알게 된 점
- **심층 방어(Defense in Depth):** 단순히 방화벽(L4) 수준에서 특정 IP를 차단하는 것을 넘어, 웹 서버(L7) 레벨에서 국가 단위의 정교한 트래픽 필터링(Geo-blocking)이 가능함을 파악함.
- **리소스 최적화:** 무분별한 해외 스캐닝 및 봇 트래픽을 입구에서 차단함으로써, 백엔드 애플리케이션 서버와 DB의 자원 낭비를 줄이고 서비스 가용성을 높임.

## 🛠️ 3. 트러블슈팅 (Troubleshooting)
### 🔍 문제 발생 (Issue)
- **상황:** GeoIP2 모듈 설정 후 Nginx 재시작 시 지시어 인식 불가 오류 발생.
- **에러 메시지:** `nginx: [emerg] unknown directive "geoip2"`

### 💡 원인 분석 (Cause)
- Nginx 빌드 시 `ngx_http_geoip2_module`이 정적으로 포함되지 않았거나, 동적 모듈 로드 설정(`load_module`)이 `nginx.conf` 최상단에 누락되어 모듈을 인식하지 못함.

### 🚀 해결 방법 (Solution)
- 시스템에 `libmaxminddb` 라이브러리를 설치한 후, Nginx 설정 파일 최상단(Main context)에 아래 구문을 추가하여 동적 모듈을 명시적으로 로드함.
  ```nginx
  load_module modules/ngx_http_geoip2_module.so;

## 🧐 4. 오늘의 회고
### 💡 성찰 및 성장
- **습득 지식:** 전 세계 IP 대역이 국가별로 어떻게 할당되고 관리되는지 메커니즘을 이해함. 이를 단순 지식이 아닌, 실제 서비스 인프라에 적용하여 '지리적 보안 계층'을 강화하는 실무 기술을 습득함.
- **인프라 관점:** 15대 규모의 대형 인프라를 운영할 때는 단일 서버 보안보다 '입구(Gate)'에서의 트래픽 제어가 전체 시스템 부하와 보안에 얼마나 지대한 영향을 미치는지 실감함.

## 🚀 Action Plan
### 🛠️ 부족한 점 및 개선 방안
- **부족한 점:** GeoIP 데이터베이스(`.mmdb`) 파일은 수시로 업데이트되는데, 현재 수동으로 관리되고 있어 최신 IP 대역 반영이 누락될 위험이 있음.
- **개선 방안:** `geoipupdate` 공식 도구를 설치하고, 매주 특정 요일에 최신 DB를 자동 다운로드하도록 `crontab` 스케줄링을 예약하여 운영 자동화를 실현할 예정.

### 🎯 내일의 목표
- **목표:** HAProxy 로드밸런서 뒤에 위치한 Nginx 서버가 클라이언트의 실제 IP를 인식하지 못하고 LB의 내부 IP만 기록하는 현상 해결.
- **핵심 과제:** 1. HAProxy에서 `option forwardfor` 설정 확인.
    2. Nginx에서 `X-Forwarded-For` 헤더를 처리하기 위한 `set_real_ip_from` (LB IP 지정) 및 `real_ip_header` 설정 완료.
    3. 실제 클라이언트 IP 기반의 보안 정책(ACL) 재검증.
