## 📝 1. 오늘의 학습 주제 및 핵심 내용
- **주요 주제: MariaDB TDE(Transparent Data Encryption)를 이용한 데이터 암호화**
- **핵심 개념 1: TDE (투명한 데이터 암호화)**
    - 내용: 데이터베이스 엔진 레벨에서 파일 저장 시 암호화하고 읽을 때 복호화하여, 애플리케이션 수정 없이 데이터를 보호하는 기술.
- **핵심 개념 2: File Key Management**
    - 내용: 외부 키 파일을 참조하여 암호화 키를 관리하고, AES_CTR 등의 알고리즘으로 데이터 무결성을 보장함.

## 💻 2. 실습 및 구현 내용
### ✅ 실습 목표
- DB 내부의 테이블뿐만 아니라 트랜잭션 로그(Redo Log)까지 암호화하여 물리적 보안 사고(디스크 유출 등)에 대비한다.

### 🛠️ 구현 결과
# [/etc/my.cnf.d/mariadb-server.cnf]

[mariadb]
# [1] 키 관리 플러그인 로드 및 경로 지정
plugin-load-add = file_key_management
file_key_management_filename = /etc/mysql/encryption/keyfile
file_key_management_encryption_algorithm = AES_CTR

# [2] InnoDB 테이블 및 로그 암호화 활성화
# 모든 테이블을 강제로 암호화하도록 설정
innodb_encrypt_tables = ON
# 트랜잭션 로그(Redo Log)까지 암호화하여 빈틈없는 보안 구현
innodb_encrypt_log = ON

# [3] 암호화 성능 최적화
# 암호화 처리를 위한 전용 스레드 4개 할당
innodb_encryption_threads = 4
# 키 로테이션 주기 설정 (보안성 강화)
innodb_encryption_rotate_key_age = 1

## ✨ 새롭게 알게 된 점
- **투명성(Transparency):** 워드프레스(WordPress) 소스 코드를 단 한 줄도 수정하지 않고, DB 엔진 설정만으로 개인정보 보호법(저장 데이터 암호화) 요구사항을 완벽히 충족할 수 있다는 점이 매우 효율적임을 체감함.
- **로그 암호화의 중요성:** 데이터 테이블만 암호화할 경우, 트랜잭션 처리 과정에서 생성되는 Redo Log 등에 평문 데이터가 남을 수 있음. 따라서 `innodb_encrypt_log` 설정을 통해 데이터의 생명주기 전체를 보호해야 함을 깨달음.

## 🛠️ 3. 트러블슈팅 (Troubleshooting)
### 🔍 문제 발생 (Issue)
- **상황:** TDE 설정 적용 후 MariaDB 서비스를 재시작했으나, 서비스 구동에 실패하고 프로세스가 즉시 종료됨.
- **에러 메시지:** `[ERROR] InnoDB: Encryption can't open encryption key file (errno: 13 "Permission denied")`

### 💡 원인 분석 (Cause)
- `/etc/mysql/encryption/keyfile`을 생성할 당시 `root` 계정으로 작업하여 파일 소유권이 `root`에 있었음.
- MariaDB 프로세스는 `mysql` 사용자 권한으로 실행되므로, 보안 키 파일에 접근(Read)할 권한이 없어 암호화 모듈을 로드하지 못한 것임.

### 🚀 해결 방법 (Solution)
- **소유권 및 권한 변경:** 키 파일을 포함한 디렉토리 전체의 소유권을 `mysql`로 변경하고, 외부 유출 방지를 위해 권한을 600(소유자 읽기/쓰기만 허용)으로 제한하여 해결함.
  ```bash
  # 소유권 변경 (디렉토리 포함)
  chown -R mysql:mysql /etc/mysql/encryption
  
  # 키 파일 보안 권한 강화 (600: rw-------)
  chmod 600 /etc/mysql/encryption/keyfile
  
  # 서비스 재시작
  systemctl restart mariadb
---
