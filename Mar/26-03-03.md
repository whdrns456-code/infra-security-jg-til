📝 1. 오늘의 학습 주제 및 핵심 내용
주요 주제: MariaDB Master-Slave 복제 구축 및 권한 보안 설정

핵심 개념 1: Master-Slave Replication

내용: 바이너리 로그(Binary Log)를 기반으로 마스터의 변경 사항을 슬레이브가 실시간 동기화하는 구조. Slave_IO_Running과 Slave_SQL_Running이 모두 Yes여야 정상 작동함.

핵심 개념 2: RBAC (Role-Based Access Control) 및 read-only 설정

내용: 복제 전용 계정(REPLICATION SLAVE)과 서비스용 계정(ALL PRIVILEGES)을 분리하고, 슬레이브 서버에 read-only 옵션을 적용하여 데이터 무결성을 확보함.

💻 2. 실습 및 구현 내용
✅ 실습 목표
MariaDB 기반 DB 이중화 구성 및 WordPress 연동을 위한 인프라 환경 구축

🛠️ 구현 결과
SQL
-- [Master 서버] 복제 전용 계정 생성 및 권한 부여
CREATE USER 'chill_slave'@'192.168.100.51' IDENTIFIED BY '1234';
GRANT REPLICATION SLAVE ON *.* TO 'chill_slave'@'192.168.100.51';
FLUSH PRIVILEGES;

-- [Slave 서버] 복제 시작 및 상태 확인
STOP SLAVE;
CHANGE MASTER TO 
  MASTER_HOST='192.168.100.50',
  MASTER_USER='chill_slave',
  MASTER_PASSWORD='1234',
  MASTER_LOG_FILE='mysql-bin.000003',
  MASTER_LOG_POS=342;
START SLAVE;
SHOW SLAVE STATUS\G
✨ 새롭게 알게 된 점
Root의 특권: read-only = 1 설정이 되어 있어도 SUPER 권한을 가진 Root 계정은 쓰기 작업이 가능하다는 점을 확인. 실제 서비스 계정으로 테스트해야 정책 검증이 정확함.

Host 식별: MariaDB는 User@Host를 하나의 쌍으로 인식하므로, 네트워크 대역(192.168.100.%) 설정 시 접속 주소 일치가 필수적임.

🛠️ 3. 트러블슈팅 (Troubleshooting)
🔍 문제 발생 (Issue)
상황: 슬레이브 서버에서 Last_IO_Error: Access denied 발생하며 복제가 중단됨.

에러 메시지: Last_IO_Errno: 1597, Errno: 1045

💡 원인 분석 (Cause)
마스터에서 생성한 계정명(chill_slave)과 슬레이브에서 접속 시 설정한 계정명(chill)이 불일치함.

복제 전용 권한(REPLICATION SLAVE) 누락으로 인한 인증 거부.

🚀 해결 방법 (Solution)
마스터에서 계정명과 권한을 재정의하고, 슬레이브에서 CHANGE MASTER TO 구문을 통해 MASTER_USER 정보를 올바르게 수정 후 START SLAVE 재실행.
