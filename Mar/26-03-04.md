📝 1. 오늘의 학습 주제 및 핵심 내용
주요 주제: SSL/TLS 프로토콜의 이해 및 암호화 통신 원리

핵심 개념 1: SSL/TLS Handshake

내용: 클라이언트와 서버가 서로를 인증하고, 데이터를 암호화하기 위한 대칭키(세션 키)를 안전하게 공유하는 과정입니다.

핵심 개념 2: 공개키(비대칭키) vs 대칭키 방식

내용: 보안성과 성능을 모두 잡기 위해, 키 교환 단계에서는 공개키 방식을 사용하고, 실제 데이터 전송 단계에서는 연산 속도가 빠른 대칭키 방식을 혼합하여 사용합니다.

💻 2. 실습 및 구현 내용
✅ 실습 목표
OpenSSL을 이용한 자가 서명 인증서(Self-signed Certificate) 생성 및 Nginx HTTPS 설정

🛠️ 구현 결과
Bash
# 1. 개인키(Private Key) 생성
openssl genrsa -out server.key 2048

# 2. CSR(인증서 서명 요청서) 생성
openssl req -new -key server.key -out server.csr

# 3. 자가 서명 인증서(CRT) 생성 (유효기간 365일)
openssl x509 -req -days 365 -in server.csr -signkey server.key -out server.crt

# 4. Nginx 설정 예시
server {
    listen 443 ssl;
    ssl_certificate /etc/nginx/ssl/server.crt;
    ssl_certificate_key /etc/nginx/ssl/server.key;
}
✨ 새롭게 알게 된 점
**CA(Certificate Authority)**의 역할: 브라우저가 신뢰하는 제3의 기관이 서명한 인증서여야만 '주의 요함' 경고가 뜨지 않는다는 점을 확인했습니다.

SNI(Server Name Indication): 하나의 IP 주소에서 여러 SSL 인증서를 호스팅할 수 있게 해주는 기술적 원리를 이해했습니다.

🛠️ 3. 트러블슈팅 (Troubleshooting)
🔍 문제 발생 (Issue)
상황: HTTPS 설정 후 브라우저 접속 시 '연결이 비공개로 설정되어 있지 않습니다' 경고 발생

에러 메시지: NET::ERR_CERT_AUTHORITY_INVALID

💡 원인 분석 (Cause)
공인 인증 기관이 아닌, 로컬 환경에서 직접 만든 Self-signed Certificate를 사용했기 때문에 브라우저가 신뢰할 수 없는 인증서로 판단함.

🚀 해결 방법 (Solution)
실습 환경에서는 브라우저의 '고급' 설정을 통해 강제 접속하거나, 운영 환경에서는 Let's Encrypt와 같은 무료 공인 CA를 통해 인증서를 발급받아야 함을 인지함.
