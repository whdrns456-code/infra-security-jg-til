📝 1. 오늘의 학습 주제 및 핵심 내용
주요 주제: 웹 애플리케이션 아키텍처 및 보안 기초 환경 구축

핵심 개념 1: 시스템 침투와 권한 상승

LPE (Local Privilege Escalation): 로컬 환경 내에서 낮은 권한을 가진 사용자가 관리자 권한을 획득하는 기법.

RCE (Remote Code Execution): 원격지에서 대상 시스템에 코드를 실행시키는 공격. 버퍼 오버플로우 등을 통해 발생함.

핵심 개념 2: 레이스 컨디션 (Race Condition)

공유 자원에 여러 프로세스/쓰레드가 동시에 접근할 때 발생하는 실행 순서의 불일치 문제.

TOCTU (Time-of-Check to Time-of-Use): 보안 검사 시점과 실제 자원 사용 시점 사이의 간극을 악용하는 대표적인 레이스 컨디션 공격 기법.

💻 2. 실습 및 구현 내용
🏗️ 웹 애플리케이션 아키텍처 이해
현대적인 웹 서비스는 단순한 서버-클라이언트 모델을 넘어 다중 계층 구조를 가집니다.

Client Side: 브라우저(Firefox, Edge, Chrome)가 HTML, CSS, JavaScript를 해석하는 영역 (Client Side Script).

Web Server: Apache(httpd), Nginx 등 정적 컨텐츠를 처리하고 클라이언트의 요청을 받음.

WAS (Web Application Server): PHP, JSP, ASP 등 서버 사이드 스크립트가 실행되어 비즈니스 로직을 처리하는 영역.

Database: MariaDB, MySQL 등 실제 데이터가 테이블 형태로 저장되고 SQL을 통해 조회/수정되는 영역.

🛠️ DVWA(Damn Vulnerable Web Application) 환경 구축 (Rocky Linux)
보안 실습을 위한 취약한 웹 서버 환경을 직접 구축하였습니다.

패키지 설치: httpd, mariadb-server, php, php-mysqli 등 LAMP 스택 구성.

DB 보안 설정: mariadb-secure-installation을 통한 root 암호 설정 및 익명 사용자 삭제.

애플리케이션 설정: git clone으로 소스 복사 후 config.inc.php에서 DB 접속 정보(User: dvwa, PW: p@ssw0rd) 수정.

권한 관리: 웹 서버 데몬이 정상 작동하도록 /var/www/html 내 파일 소유권을 apache 계정으로 변경.

🛠️ 3. 트러블슈팅 (Troubleshooting)
🔍 문제 발생 (Issue)
상황: Apache 서버 구동 후 PHP 페이지 접속 시 소스 코드가 그대로 노출되거나 500 에러 발생.

💡 원인 분석 (Cause)
PHP 모듈이 Apache와 정상적으로 연동되지 않았거나, DirectoryIndex 설정에 index.php가 누락되어 우선순위 밀림.

🚀 해결 방법 (Solution)
/etc/httpd/conf/httpd.conf 수정: DirectoryIndex index.php index.html 설정.

추가 패키지 설치: php-fpm 및 php-mysqlnd 설치 후 서비스 재시작.

phpinfo(); 함수를 담은 테스트 파일을 통해 연동 상태 최종 확인.

🧐 4. 오늘의 회고
💡 성찰 및 성장
습득 지식: 단순한 해킹 툴 사용법이 아니라, 요청(Request)과 응답(Response)이 오가는 HTTP 프로토콜의 구조와 웹 서버-WAS-DB로 이어지는 인프라의 흐름을 이해하는 것이 보안의 시작임을 깨달았습니다.

협업/소통: Burp Suite와 같은 프록시 도구를 설정하며 클라이언트와 서버 사이의 통신을 가로채는 원리를 시각적으로 확인했습니다.

🚀 Action Plan
부족한 점: 리눅스 권한(SetUID, SetGID) 악용 사례에 대한 실습 부족.

개선 방안: 내일은 실제 DVWA의 Security Level을 조정하며 단계별 SQL Injection 시나리오를 연습할 예정.

내일의 목표: HTTP 메서드(GET/POST/PUT/DELETE)별 취약점 분석 및 Burp Suite를 활용한 패킷 변조 숙달.
