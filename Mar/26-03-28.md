## 📝 1. 오늘의 학습 주제 및 핵심 내용
- **주요 주제: 리눅스 커널 방화벽(IPTables)의 정교한 정책 제어**
- **핵심 개념 1: IPTables 규칙의 우선순위 (Top-Down 방식)**
    - 규칙은 위에서부터 순차적으로 적용됨.
    - `INPUT DROP` 정책을 먼저 적용하면 이후의 허용 규칙이 무시되므로, 반드시 **관리용 포트(SSH) 허용 규칙을 최상단(`-I`)에 배치**해야 함.
- **핵심 개념 2: 패킷 처리 방식의 차이 (DROP vs REJECT)**
    - **DROP:** 패킷을 조용히 버림. 클라이언트는 타임아웃(Timeout)이 발생할 때까지 기다려야 하므로 스캔 공격 속도를 늦추고 시스템 존재를 숨김.
    - **REJECT:** '거절' 메시지를 응답(ICMP Destination Unreachable). 클라이언트는 즉시 연결 거부를 확인 가능함.

## 💻 2. 실습 및 구현 내용
### ✅ 실습 목표
- 외부 서비스용 80포트와 관리용 22포트를 제외한 모든 접근을 차단하는 화이트리스트 기반 방화벽 구축.

### 🛠️ 구현 결과

```bash
# 1. SSH 연결 유지를 위해 22번 포트 최우선 허용 (가장 중요!)
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# 2. 루프백 및 상태 유지 패킷 허용
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 3. HTTP(80) 서비스 포트 허용
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# 4. 기본 정책을 DROP으로 변경 (나머지 모두 차단)
iptables -P INPUT DROP

# 5. 설정 결과 확인
iptables -L -v

✨ 새롭게 알게 된 점
보안성 vs 편의성: REJECT는 내부 네트워크 관리 시 오류 확인에 유리하지만, 외부망(DMZ) 접점에서는 보안을 위해 시스템의 존재를 숨기는 DROP을 사용하는 것이 표준임을 배움.

🛠️ 3. 트러블슈팅 (Troubleshooting)
🔍 문제 발생 (Issue)
상황: iptables -P INPUT DROP 실행 직후 SSH 세션이 끊기며 원격 접속 불가능 상태 발생.

에러 메시지: ssh: connect to host [IP] port 22: Connection timed out

💡 원인 분석 (Cause)
SSH 허용 규칙을 넣기 전 기본 정책(Policy)을 DROP으로 바꿨기 때문에, 현재 접속 중인 관리자 세션까지 즉시 차단됨. 즉, 나가는 문은 열려있어도 들어오는 응답이 막혀 세션이 끊긴 것.

🚀 해결 방법 (Solution)
가상머신 하이퍼바이저 콘솔(Console)에서 직접 접속하여 최상단에 규칙 삽입 명령 수행.

명령어: iptables -I INPUT 1 -p tcp --dport 22 -j ACCEPT

교훈: 원격 서버 설정 시에는 항상 -I(Insert)를 사용하여 허용 규칙을 정책 최상단에 먼저 넣는 습관이 필수적임.
